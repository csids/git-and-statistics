4.1 Concepts

Note

Read the org introduction to understand the different requirements for code, results, and data, and some suggested solutions regarding layouts.

Most analyses have three main sections:

  • Code.
  • Results.
  • Data.

Yet each of these sections have extremely different requirements.

Code should:

  • Be version controlled (I.e. available on GitHub!).
  • Be publicly accessible (sometimes).
  • Have 1 analysis pipeline that logically and sequentially details all steps of the data cleaning, analysis, and result generation.

Results should:

  • Be immediately shared with close collaborators.
  • Have each set of results saved and accessible, so that you can see how your results have changed over time (i.e. “if we run the code today, do we get similar results to yesterday?”).

Data should:

  • Be encrypted (if sensitive).
  • Not stored on the cloud (if sensitive).
Note

Determining appropriate solutions for you and your team is an incremental process that requires input from all team members regarding their particular needs. Here are some suggestions that can be used as a starting point.

4.2 The minimalist solution

4.2.1 Code

Code is stored in a private GitHub repo, with access primarily controlled via teams, not via individual accounts.

This GitHub repo does not contain:

  • Results.
  • Data.

This code contains an obvious analysis pipeline that logically and sequentially details all steps of the data cleaning, analysis, and result generation.

For example:

  • One script file:
    • Run.R
  • Multiple numbered script files:
    • 01_clean_data.R
    • 02_analyze_data.R
    • 03_create_graphs.R
  • One script file that renders a Quarto file:
    • Run.R
    • quarto/quarto.qmd
  • One script file that renders an R Markdown file:
    • Run.R
    • rmarkdown/rmarkdown.rmd

Functions will be stored in a folder:

  • functions/

4.2.2 Results

Results will be stored in a traditional cloud storage service.

4.2.3 Data

Data will be stored locally.
